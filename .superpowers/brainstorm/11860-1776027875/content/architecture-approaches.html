Watch App Architecture — 3 Approaches

How should the iPhone and Watch apps communicate and share data?

A

WatchConnectivity Only

iPhone ↔ Watch sync via WCSession. Watch stores local copy of today's data. iPhone is the source of truth.

Pros

  • Real-time sync when phone is nearby
  • Simple mental model — iPhone owns the data
  • No iCloud complexity on Watch

Cons

  • Watch data stale when phone is out of range
  • Entries logged on Watch queue until phone reconnects
  • New framework to integrate (WatchConnectivity)
B

Shared App Group + iCloud KVStore ⭐ Recommended

Both apps read/write to the same App Group container + iCloud KVStore. Same sync mechanism the Widget already uses. Watch has full independent access.

Pros

  • Builds on existing persistence architecture
  • Watch works fully independently
  • Widget, iPhone, and Watch all share one data path
  • No new sync framework needed
  • iCloud handles cross-device sync automatically

Cons

  • iCloud KVStore has 1MB limit (fine for hydration data)
  • Sync latency (~seconds, not instant)
  • Need conflict resolution (you already have timestamp-based)
C

WatchConnectivity + App Group Hybrid

WatchConnectivity for real-time updates (instant sync). App Group + iCloud as fallback when phone is away. Best sync speed, most complex.

Pros

  • Instant sync when phone is nearby
  • Independent operation when phone is away
  • Best possible user experience

Cons

  • Two sync paths = more complexity and edge cases
  • Conflict resolution across both channels
  • Significantly more code to maintain
  • Overkill for hydration data (not time-critical)
